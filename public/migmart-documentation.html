01App Overview

MigMart is a loyalty-enabled grocery shopping platform. It simulates a real-world retail experience where customers earn points on purchases, spin a daily prize wheel, complete challenges, unlock badges, and redeem points for discounts and free items — all within a polished, mobile-friendly web app.

The platform is designed for a project presentation and uses a mix of real backend infrastructure (Supabase for auth, profiles, and event logging) with client-side localStorage simulation for features like challenge progress, referral history, and checkout counts that would normally require a more complex backend.

03Architecture & Routing

/auth/callback Server (Route Handler) Exchanges Supabase email verification code for a session
/onboarding Server + Client First-time setup: display name, preferences
/ Server + Client Main storefront — requires authentication
/admin Server Admin dashboard — requires auth + is_admin flag + session cookie

04Authentication & Onboarding

Authentication is handled entirely by Supabase Auth using email + password. Google and Apple OAuth were deliberately removed in favour of a clean, simple email-only flow.

Sign Up Flow

Enter name, email, password
→
Supabase sends verification email
→
User clicks link → /auth/callback
→
Session created → /onboarding

Sign In Flow

Enter email + password
→
Supabase validates credentials
→
Profile fetched → / or /onboarding
Email verification redirect: Set to https://migmart.vercel.app/auth/callback — the callback route exchanges the one-time code for a real session cookie and upserts the user profile before redirecting to onboarding.

Session & Security

Sessions are managed via @supabase/ssr cookie-based tokens, handled server-side. Protected routes check for a valid session using createSupabaseServerClient() and redirect to /auth if none is found. Sign-out is guarded by a confirmation modal so users don't accidentally log out.

05Storefront

The main page (/) is the storefront — a server-rendered page that loads the user's profile, leaderboard, and achievements from Supabase, then passes them as props to the Storefront client component.

The storefront is organised into four tabs:

🛍️
Shop
Browse and filter products by category or keyword. Add to cart, view discounts, and checkout.
🏆
Challenges
Active and weekly challenges that award bonus points when completed. Progress tracked in localStorage.
🎁
Rewards
Redeem earned points for discounts (GHC 5–25 off) or free grocery items.
🔗
Referral
Share your unique code with friends. Simulated referral tracking via localStorage.

Product Catalogue

Products are defined statically in lib/products.ts across 6 categories: Fruits & Veggies, Dairy & Eggs, Beverages, Snacks, Bakery, and Household. Each product has a name, price (in GHS), optional discount %, image, rating, and unit size. Prices are displayed using the Ghanaian Cedi format (e.g. GH₵ 5.84).

Shopping Cart

The cart is managed by a Zustand store (lib/store.ts) — entirely in-memory. The cart drawer slides in from the right, supports quantity adjustments and item removal, and shows the live subtotal. Checkout is guarded by a confirmation modal showing the total and the points the user will earn.

06Loyalty & Points System

Points are the core currency of MigMart's engagement layer. Every action a user takes can award points, which accumulate towards tier upgrades and can be redeemed for rewards.

How Points Are Earned

Action Points Awarded
Checkout 1 pt per GHC 1 spent (e.g. GHC 45 order → 45 pts)
Daily Spin 25 – 160 pts depending on wheel segment
Completing a Challenge 50 – 100 pts per challenge
Referring a friend 50 pts per successful referral

Loyalty Tiers

Tier Points Required Benefits
🥉 Bronze 0 pts Access to spin wheel and basic rewards
🥈 Silver 700 pts Silver badge unlocked; higher reward eligibility
🥇 Gold 1,600 pts Gold badge unlocked
💎 VIP 3,000 pts VIP badge unlocked; full reward catalogue access

Tier progress is shown in the Loyalty Vault with a progress bar and a label showing how many points remain to the next tier. Points and tiers are stored in Supabase and reflected in real time.

How It Works

A random segment is chosen. The wheel calculates the exact rotation delta needed (accounting for accumulated prior spins) so the pointer arrow always stops at the centre of the winning segment. The spin animation runs for 4 seconds with a custom cubic-bezier easing for a realistic deceleration. Points are written to Supabase on completion.

Spins are limited by the spins_available field in the user's profile. Once spins run out, the button is disabled.

11Referral System

Each user gets a unique referral code generated from their user ID and display name. Referrals are simulated via localStorage (lib/referrals.ts) for presentation purposes.

How It Works

User copies their code
→
Enters a friend's name to simulate invite
→
50 pts awarded + referral logged
→
Referral appears in history list

In a production system, the referral code would be tied to a sign-up flow, validated server-side, and the points awarded only after the friend completes their first purchase.

12Leaderboard

A ranked list of top shoppers by points, fetched server-side from Supabase profiles. Key UX details:

  • Top 3 receive medal icons (🥇 🥈 🥉)
  • The current user is labelled "(you)" for context
  • A contextual hint shows how many points the user is behind the next rank
  • Users can opt out of visibility with the eye icon — their entry is hidden from other users

13Admin Dashboard

A protected admin area accessible at /admin. Access requires two checks to pass:

  1. A valid admin_session httpOnly cookie set by the password gate (/admin/login)
  2. The user's Supabase profile must have is_admin = true

The admin password is stored as an environment variable (ADMIN_PASSWORD) and never exposed to the client. The session cookie expires after 8 hours.

The admin dashboard (rendered by components/admin-dashboard.tsx) provides an overview of platform metrics and user activity.

14Data Layer

Persistent data lives in Supabase Postgres. The following tables are used:

Table Key Columns Purpose
profiles user_id, display_name, points, tier, spins_available, is_admin, onboarding_completed Core user state
user_events user_id, event_type, payload Audit log of all significant actions
spin_events user_id, reward_points, reward_type, metadata Records each wheel spin outcome
point_ledger user_id, points_change, reason, metadata Double-entry log of all point transactions
user_achievements user_id, achievement_key, unlocked_at Unlocked badge records
achievements key, title, description Badge definitions reference table
